# Documentation Maintenance Guide

This guide defines the repository documentation standard for README files, operator runbooks, generated status pages, and claim-boundary materials.

## Documentation source-of-truth order

When documents disagree, resolve them in this order:

1. `qa/mainnet-authorization-certificate.json` for technical readiness, deployment authorization, Ethereum Mainnet authorization, and Mainnet deployment status.
2. Deployment, verification, and evidence manifests generated by the relevant scripts for concrete addresses, transaction hashes, constructor arguments, and verification status.
3. Schema files under `schemas/` for required evidence shape and validation rules.
4. Generated public status documents such as `README.md`, `START_HERE.md`, `docs/CURRENT_STATUS.md`, and `docs/MAINNET_AUTHORIZATION_CERTIFICATE.md`.
5. Narrative guides, FAQs, onboarding docs, and examples.

Narrative documentation must summarize higher-priority evidence; it must not create deployment, authorization, or verification claims on its own.

## Update checklist

Use this checklist for any README, runbook, website copy, or documentation update:

- Preserve the current Mainnet claim boundary: Ethereum Mainnet authorized for manual gated deployment, Ethereum Mainnet deployed `NO`, unless real `chainId=1` transaction evidence changes the certificate and generated docs.
- Preserve the verification claim boundary: contract verified `YES` requires source/bytecode verification evidence or confirmed already-verified status.
- Keep Mainnet broadcast local-only; do not document any CI path that can deploy Ethereum Mainnet.
- Keep runtime secrets and private operator inputs out of committed docs. Refer to private templates and local-only environment files instead.
- Preserve the canonical AGIALPHA token boundary and do not describe a new AGIALPHA token deployment path on Ethereum Mainnet.
- Prefer wrapper commands from `package.json` over raw scripts in public docs so operators inherit current guardrails.
- Update `docs/DOCUMENTATION_INDEX.md` when adding or retiring a durable documentation entry point.
- Update `docs/REPOSITORY_MAP.md` when adding or moving a top-level directory, primary README, or core reviewer path.
- Run the appropriate validation commands below before committing.

## Required validation by change type

| Change type | Minimum checks |
|---|---|
| Generated public status, certificate summary, or README status text | `npm run docs:status` then `npm run assert:public-status` |
| Deployment, verification, Mainnet, Sepolia, or evidence docs | `npm run deployment:claim-boundary:check` |
| Contributor, security, or governance docs | `npm run repo:all` when practical; otherwise run the most relevant repository safety checks and document the reason |
| Website or public marketing copy | `npm run deployment:claim-boundary:check` and a manual safe-claims review against `docs/SAFE_CLAIMS_AND_TOKEN_BOUNDARY_v3_0.md`; the checker scans public copy paths such as `app/`, `content/`, `site/`, `site-assets/`, and `templates/` |
| Schema-linked docs or evidence templates | Relevant schema validation plus `npm run qa:manifest:verify` when manifests are touched |

## Versioning and generated-doc hygiene

- Do not hand-edit generated status values when the generator should be used.
- If a generated doc changes, include the generator command in the PR notes.
- Keep historical versioned docs for audit traceability unless maintainers explicitly retire them.
- Mark retired docs as superseded and link the current entry point instead of silently deleting context.
- Do not fabricate missing PDFs, deployment receipts, verification receipts, audit approvals, or governance approvals.

## PR documentation requirements

Every documentation PR should state:

- Commands run and whether they passed or failed.
- Generated artifacts, if any.
- Whether the change affects public claims, deployment authorization, verification status, or Mainnet gates.
- If no tests were run, why the change is documentation-only and which claim-boundary review was performed.
