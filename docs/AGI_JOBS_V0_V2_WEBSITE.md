## Governed runtime

A successful local proof cycle:

1. seals a bounded Work Constitution;
2. admits declared identities and rights boundaries;
3. reserves a demonstration-only economic envelope;
4. receives bids from twelve agent guilds;
5. exposes admissibility, weighted ranking, and the non-dominated Pareto frontier;
6. constitutes a five-role execution council plus two reserves;
7. compiles an eight-package proof-producing work graph;
8. records bounded execution and a source/claim graph;
9. seals commitments from a risk-sized three-, five-, seven-, or nine-seat validator parliament;
10. reveals verdicts while preserving independent dissent;
11. evaluates six separated guardian functions and dispute conditions;
12. compiles revocable memory candidates and a proof-conditioned settlement constitution;
13. seals a twenty-four-artifact SHA-256 Evidence Docket;
14. stops at `HUMAN_SETTLEMENT_REVIEW` with `NONE_GRANTED` authority.

The experience exports an Evidence Docket v3 JSON and Executive Review Brief. No generated proof is treated as permission.

## Fail-closed behavior

Seven incident rehearsals exercise all approved terminal states:

- ordinary completion → `HUMAN_SETTLEMENT_REVIEW`;
- budget rupture → `HUMAN_REVIEW_REQUIRED`;
- license conflict → `DISPUTE_OPEN`;
- identity drift, evidence tampering, validator capture, or goal drift → `SAFE_HOLD`.

No terminal state authorizes production, user funds, settlement, external action, or public reliance.

The browser experience performs:

- no model or API call;
- no RPC request;
- no wallet connection;
- no ENS lookup;
- no token or fund movement;
- no external action;
- no production execution;
- no persistent browser storage;
- no autonomous settlement or credential issuance.

## Additive integration contract

The feature is generated only into the temporary GitHub Pages `site/` artifact. It does not alter `website/v86_actual_site/**`, Solidity, Mainnet deployment evidence, Public Proof Missions, package manifests, lockfiles, or prior feature source.

The builder may update only these already-generated shared surfaces:

- `index.html`
- `routes.json`
- `sitemap.xml`
- `site-status.json`
- installed companion manifests whose shared-surface hashes must be reconciled

All other pre-existing generated files must remain byte-identical. The verifier compares a complete prebuild snapshot, rejects removals and undeclared changes, validates every emitted hash, and re-verifies companion manifests after reconciliation.
